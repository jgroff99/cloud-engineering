# AWS Security Architecture ‚Äî Week 7

## Overview

This document describes the security posture of the cloud-engineering AWS account as of Week 7. It covers IAM design, encryption, secrets management, monitoring, and what a production-hardened version of this architecture would look like. Security is treated as a first-class engineering concern, not an afterthought.

## IAM Design

### Users and Roles

| Principal | Type | Purpose |
|-----------|------|---------|
| `admin` | IAM User | Day-to-day AWS access with MFA enforced |
| `ec2-ssm-role` | IAM Role | EC2 instance profile for SSM Session Manager and CloudWatch agent access |

**Design decisions:**
- Root account MFA is enabled. Root credentials are not used for any day-to-day operations ‚Äî only for account-level tasks that explicitly require it.
- Stale lab users (`TestUser`, `policy-test-user`) were identified during a Week 7 security audit and deleted.
- EC2 instances access AWS services via instance profiles (IAM roles), never via hard-coded credentials. This is the standard pattern for all compute in the account.

### Policy Types in Use

- **Identity-based policies** are the primary mechanism for granting permissions to IAM users and roles.
- **Resource-based policies** are used on S3 buckets for cross-account or public access scenarios (e.g., static website hosting).
- **Permissions boundaries** are not currently applied but would be required in a production multi-team environment to cap what delegated administrators can grant.
- **Session policies** are available when scoping down role assumptions programmatically via STS.

### IAM Evaluation Logic

All access decisions follow this order:
1. Explicit deny anywhere (SCP, identity policy, resource policy, permissions boundary) ‚Üí **denied, no exceptions**
2. Within the same account: explicit allow on either identity policy or resource policy ‚Üí **allowed**
3. Cross-account: explicit allow required on **both** the resource policy and the identity policy ‚Üí **allowed**
4. No explicit allow ‚Üí **implicit deny**

## Encryption

### KMS Strategy

| Key Type | Usage | Status |
|----------|-------|--------|
| AWS managed key (`aws/secretsmanager`) | Secrets Manager default encryption | Active |
| AWS managed key (`aws/ssm`) | Parameter Store SecureString encryption | Active |
| Customer managed key (`week7-lab-key`) | Lab CMK for envelope encryption practice | Deleted ‚Äî scheduled deletion completed |

**Why CMKs matter in production:** Customer managed keys provide full audit trails in CloudTrail (every encrypt/decrypt call is logged), allow key policy customization to restrict which principals can use the key, and support manual rotation schedules. AWS managed keys provide encryption with no management overhead but less control. The appropriate choice depends on compliance requirements.

### Envelope Encryption

All KMS-integrated services use envelope encryption:
1. A data key is generated by KMS for each encryption operation
2. The plaintext data key encrypts the actual data locally
3. The plaintext data key is immediately discarded
4. The encrypted data key is stored alongside the ciphertext
5. Decryption reverses this: KMS unwraps the data key, the data key decrypts the data, the plaintext data key is discarded again

The KMS CMK never touches the actual data ‚Äî it only ever encrypts and decrypts data keys. This allows efficient encryption of arbitrary data sizes while keeping key usage auditable.

## Secrets Management

### Secrets Manager

Database credentials and sensitive application secrets are stored in AWS Secrets Manager.

**Configuration:**
- Secrets encrypted with the default AWS managed KMS key (`aws/secretsmanager`)
- Automatic rotation enabled via Lambda rotator function
- Rotation tested and verified ‚Äî Lambda requires VPC configuration and appropriate endpoint access to reach the target database

**Why Secrets Manager over Parameter Store for credentials:**
- Native automatic rotation with no custom scheduling logic required
- Native RDS/Aurora integration ‚Äî AWS provides the rotation Lambda out of the box
- Per-secret access control via resource-based policies
- Full CloudTrail audit trail of every secret retrieval

### Parameter Store

Non-sensitive configuration values (environment names, ARNs, feature flags) use SSM Parameter Store Standard tier, which is free. SecureString parameters use KMS encryption for values that need protection but don't require rotation.

## Monitoring and Audit

### CloudTrail

A trail (`cloud-engineering-trail`) is configured to capture API activity across the account, with logs delivered to S3 (`cloudtrail-logs-ACCOUNT_ID`). Logging is currently paused to manage S3 storage costs during the learning phase.

**What CloudTrail captures:**
- Every API call: action, identity, timestamp, source IP, request parameters, success/failure
- Management events (control plane): creating/deleting resources, IAM changes, KMS key usage
- Used for forensic investigation ‚Äî answers "what happened and who did it"

**In production:** CloudTrail would be enabled continuously with log file validation, delivered to a separate security account's S3 bucket, and monitored via CloudWatch metric filters and alarms for high-priority events (root login, IAM policy changes, security group modifications).

### GuardDuty

GuardDuty was studied and configured during Week 7. It provides continuous threat detection by analyzing CloudTrail logs, VPC Flow Logs, and DNS logs using ML and AWS threat intelligence feeds.

**What GuardDuty detects that CloudTrail cannot:**
- Credentials used from unusual geographic locations or Tor exit nodes
- EC2 instances performing port scanning or cryptocurrency mining
- Brute force attempts against EC2 instances
- Compromised instance behavior patterns

GuardDuty complements CloudTrail: CloudTrail is the written record, GuardDuty is the analyst reading the record in near-real-time.

A full account security audit was performed using AWS-native tools:

| Tool | Findings | Resolution |

## Cross-Account and Organization Patterns

### STS and Temporary Credentials

All compute services (EC2, Lambda) receive temporary credentials via STS. The service assumes the attached execution role on behalf of the resource ‚Äî no `AssumeRole` call is required in application code. The AWS SDK automatically reads credentials from environment variables injected at runtime.

Temporary credentials always consist of three components: access key ID, secret access key, and session token. All three are required for authenticated API calls.

### SCPs in a Production Organization

In a production AWS Organization, Service Control Policies would be applied to enforce account-level guardrails:

```
Root
‚îú‚îÄ‚îÄ Security OU
‚îÇ   ‚îî‚îÄ‚îÄ [No SCP restrictions beyond org defaults]
‚îú‚îÄ‚îÄ Production OU
‚îÇ   ‚îú‚îÄ‚îÄ Deny: leaving the organization
‚îÇ   ‚îú‚îÄ‚îÄ Deny: disabling CloudTrail
‚îÇ   ‚îú‚îÄ‚îÄ Deny: disabling GuardDuty
‚îÇ   ‚îî‚îÄ‚îÄ Restrict: allowed regions (deny all except us-east-1, us-east-2)
‚îî‚îÄ‚îÄ Sandbox OU
    ‚îú‚îÄ‚îÄ Deny: production-scale resource types
    ‚îî‚îÄ‚îÄ Restrict: maximum instance sizes
```

**Key SCP principles:**
- SCPs are not grants ‚Äî an SCP allow does not give anyone permission, it only defines the boundary of what's possible. IAM policies must still explicitly grant access.
- Explicit deny in an SCP overrides any IAM allow, with no exceptions.
- Global services (IAM, STS, CloudFront, Route 53) must be exempted from region restriction deny lists, otherwise accounts break.
- `LeaveOrganization` denial belongs at the Root level to prevent any account from escaping governance controls.

## Security Posture Summary

| Domain | Current State | Production Recommendation |
|--------|--------------|--------------------------|
| Root account | MFA enabled, no access keys | No change needed |
| IAM users | Single admin user with MFA | Add IAM Identity Center for SSO |
| Compute credentials | Instance profiles on all EC2 | No change needed |
| Encryption at rest | AWS managed keys on all services | Add CMKs for regulated data |
| Secrets rotation | Enabled via Secrets Manager | No change needed |
| CloudTrail | Configured, logging paused | Enable continuously, ship to security account |
| Threat detection | GuardDuty studied and configured | Enable continuously in production |
| Network exposure | No public SSH, SGs reviewed | Add VPC Flow Logs for network audit trail |
| Organization controls | Single account | Implement multi-account with SCPs for production |
